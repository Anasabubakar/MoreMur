01Executive Summary

PRODUCT OVERVIEW

Murmur is an anonymous, organisation-scoped discussion and community intelligence platform. It gives employees, students, and fellows a dedicated space to post thoughts, reactions, gossip, feedback, and opinions, completely separated from professional channels like Slack, WhatsApp, or Discord, where such content creates noise and violates community guidelines.

Every user is Anonymous. No display names. No avatars. No public profiles. The only identity that matters is the organisation a user belongs to, verified by their institutional email address. Murmur is built for exclusivity — each organisation gets its own isolated feed, enforced at the infrastructure level.

The Problem

Organisations running communities through WhatsApp groups, Discord servers, or Slack workspaces face a persistent challenge: members use those channels for gossip, drama, venting, and off-topic arguments. This:

  • Pollutes professional or learning-focused channels
  • Violates community guidelines and creates moderation overhead
  • Suppresses authentic opinions due to fear of social consequences
  • Buries important announcements and discussions

The Solution

Give that energy its own home. Murmur is where the gossip lives. By separating this behaviour into its own platform, organisations preserve the integrity of their main channels while still giving their communities a legitimate, structured outlet.

03Core Product Concept

HOW MURMUR WORKS

The Anonymous Identity Model

Every user who signs up on Murmur is assigned the identity: Anonymous. No username. No avatar. No bio. No public profile. Even the user themselves cannot view their own profile page. Internally, Murmur maintains a hidden identity record for each user — accessible only to platform administrators, never exposed under any circumstances. It exists solely for moderation, abuse prevention, and data compliance.

Organisation Isolation

Murmur enforces strict feed isolation by organisation. Users must sign up with an institutional or company email address, which receives a one-time OTP for verification.

  • Generic email providers are blocked at signup: gmail.com, yahoo.com, outlook.com, and similar are rejected with a clear error message.
  • OTP verification ensures the person actually has access to that institutional inbox, preventing impersonation.
  • Once verified, the user is permanently scoped to that organisation's feed. Feeds never cross.

Post Lifecycle

  1. User writes a Murmur: text, a reaction, a question, or a poll.
  2. Post is published to the organisation's feed with the author shown as Anonymous.
  3. Other users in the same organisation can Like, Comment, React, or Repost.
  4. Comments and reposts are also anonymous.
  5. Posts can be reported. Reports are reviewed by Murmur admins only.

Data Ethics Guardrails

All data sold or reported at the Enterprise tier must comply with strict anonymisation standards:

  • Fully anonymised — no individual can be identified from any data point.
  • Aggregated — minimum cohort size of 10 per data point to prevent inference attacks.
  • Consented — users are informed on signup that anonymised aggregate data may be used for insights.
  • Compliant — NDPA (Nigeria Data Protection Act) and GDPR-adjacent standards.

Data Model: Key Entities

Users

  • id (UUID, internal only)
  • email (hashed, stored encrypted, never exposed via public API)
  • org_id (foreign key to Organisations)
  • status: active | suspended | shadow_banned
  • created_at, last_active_at

Organisations

  • id, name, domain
  • tier: free | pro | enterprise
  • logo_url, primary_color
  • status: pending | active | suspended

Posts (Murmurs)

  • id, org_id, user_id (internal only), content, category_tag
  • like_count, comment_count, repost_count
  • status: published | removed | shadow_removed
  • created_at

Anonymity Architecture

The anonymity guarantee is enforced at the database query layer, not just the API layer:

  • user_id is never returned in any public-facing API response.
  • Join queries that would link a post to a user record are forbidden in the public API scope.
  • Admin queries are isolated to a separate internal API, protected by separate authentication with full audit logging.
  • All admin access to user identity records is logged with timestamp, action taken, and admin ID.

Core Design Principles

PrincipleImplementation
Thick bordersAll UI cards, buttons, inputs, and containers have a 2 to 4px solid black border.
Hard shadowsOffset box shadows (4px 4px 0px #000) replace soft drop shadows everywhere.
High contrastBlack on yellow. White on black. No gradients. No blur. No opacity tricks.
Bold typographyHeadlines in Bebas Neue / Arial Black. Body in DM Sans. Code in Space Mono.
No rounded cornersBorder radius is 0 everywhere. Sharp corners only.
Interactive statesOn hover, shadow offsets increase. On click, shadow collapses. Button physically depresses.
Visible gridThe layout grid is almost visible. Blocks feel deliberately placed, not floated.

Post Card Anatomy

  • Black 3px border on all sides with 4px 4px black hard shadow
  • White background, zero border radius
  • Yellow tag badge for the post category (Gossip, Rant, Opinion, etc.)
  • "ANONYMOUS" label in bold, all caps, with a randomly generated geometric icon per session
  • Post text in DM Sans 14px, max width constrained
  • Action row at the bottom: Like count, Comment count, Repost button, Report flag

09Risks & Mitigations

RISK REGISTER

Risk Likelihood Mitigation
Anonymity misused for harassment High OTP ties account to real email. Admins can de-anonymise and act. Shadow ban and suspension tools available.
Orgs reject 'gossip' framing Medium Position as 'community intelligence' and 'anonymous feedback platform' in all org-facing materials.
Low post volume kills network effect Medium Seed the Learn2Earn cohort aggressively. Add conversation starters and prompts to new feeds.
Data breach exposes user emails Low-Medium Emails are hashed and encrypted at rest. Internal admin API is fully separated from public API.
Legal exposure from toxic content Medium Clear ToS, automated moderation, active admin queue, and a formal legal takedown process.
Copycat by larger platform Low Moat is org-level data and community trust, not the feature set. Move fast and lock in orgs early.
